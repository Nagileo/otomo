# 16 · 多模态（视觉）能力路线 — 图像与视频

> 来源：2026-06-29 讨论。用户问"接入 VLM API（qwen3.6-flash 这类）后能做什么、模型能力到哪、值不值得专门做"。
> 共识：多模态不只是识番——它是 Otomo 的"**视觉入口**"层（图像 + 视频）。但 VLM 是弱入口、不是视觉权威；和文本侧一样，**识别给线索 → 回锚 Bangumi / 专用引擎的 canonical → 标 source + confidence**。本文先诚实划清"通用 VLM 自身能力到哪"，再排能做什么、按什么优先级、踩什么坑。承接 docs/15 的 D1（已落地识番基线）。

---

## 0. 定位与原则

- **视觉入口，不是视觉权威**：VLM 输出永远是 weak signal，必须回锚到 canonical（Bangumi / trace.moe / 专用引擎），绝不直接当事实进答案。
- **选型**：`qwen3.6-flash`（原生多模态，OCR / 文档解析强）主视觉；`deepseek-v4-flash` 主文本。config 已分 `llm_*` / `vlm_*`，识番工具填 `vlm_model=qwen3.6-flash` 即可，零代码改动。
- **不破坏主线**：所有视觉结果标 `source`（`image` / `image_match` / `pixiv` …）+ `confidence`，进 claim verifier 当弱证据。
- **已落地基线**：D1 `identify_acgn_screenshot`（`upload://` 服务端解析 + VLM + 回锚 Bangumi search）。

---

## 1. 先回答"通用 VLM 自身能力到哪"（你最好奇的）

基于 `qwen3.6-flash` 级别的原生多模态模型，**诚实分层**：

| 能力 | 可靠度 | 说明 |
|---|---|---|
| OCR（印刷 / 手写 / 表格 / PPT） | **强** | OmniDocBench 89.9；台词、字幕、榜单截图、杂志扫描都能转结构化（`qwenvl markdown` / `qwenvl html` 带位置信息） |
| 画面 / 物体 / 构图描述 | **强** | 画面在讲什么、色调、镜头语言 |
| 热门作品 / 角色识别 | 中 | 大热番（孤独摇滚、咒术回战）多半认得；冷门、长尾、撞画风的容易自信地认错 |
| 画风 / 流派判断 | 中 | 能说"京阿尼日常系画风"，但不精确、不可作为事实 |
| 冷门精确识番 | **弱** | 训练覆盖稀疏，倾向编一个看起来合理的答案 |
| 相似角色 / 同人 / cosplay 区分 | **弱** | 撞脸、二创、真人 cos 都难 |
| 精确到"第几集第几秒" | **不能** | 通用 VLM 给不了，这是 trace.moe 的活 |
| 同人图 / 插画溯源（出自哪） | **不能** | 要专用以图搜图引擎（SauceNAO / ascii2d） |

**结论（两条铁律）**：
1. VLM 擅长"**读图里的字 + 描述画面 + 认热门**"，不擅长"**精确溯源 + 长尾识别**"。
2. 所以：① 精确识番 / 溯源交给专用引擎，VLM 只做语义补充；② VLM 结论一律回锚 canonical，不直接当事实。

> 这也反过来证明：现在 D1 的"VLM 弱入口 + 回锚 Bangumi"架构是对的——它不依赖 VLM 报准番名，VLM 报不准也能靠回锚兜底。

## 2. 工作线（按"铺什么视觉能力"组织）

| 线 | 铺什么 | 子项 |
|---|---|---|
| **I 识别入口** | 更强的视觉 entry | I1 trace.moe 双路识番 · I2 角色识别→声优网络 · I3 多图 + 置信度校准 |
| **O OCR 能力族** | 把"图里的字"变检索入口 | O1 台词 / 字幕 OCR→定位 · O2 榜单 / 杂志 / PPT→结构化 |
| **F 视觉特征** | 画风 / 构图→推荐 | F1 画风描述→相似作品入口 |
| **S 搜图 / 溯源** | 同人图 / 插画溯源与推荐 | S1 以图搜图（SauceNAO / ascii2d）· S2 pixiv 图像推荐 |
| **V 视频多模态** | 补 D2 缺的"画面层" | V1 抽帧 + OCR（无字幕视频）· V2 关键帧识番 |

## 3. 各项详述

### I1 · trace.moe 双路识番（识番主力升级）
**目标**：动画截图精确到 **番剧 + 集数 + 时间戳**。
**现状**：D1 已有 VLM 识番（弱判断）。
**做法**：截图先走 **trace.moe**（基于 AniList 的动画截图反查）→ 拿到番剧 + 集 + 秒 → 回锚 Bangumi（AniList↔Bangumi 映射，推荐侧已有 external mapping）；VLM 并行做"角色是谁 / 画面在讲什么"语义补充。两路结果 merge，各标 confidence。
**能力边界**：trace.moe 只认 **TV 动画截图**（番剧库内）；galgame CG / 漫画 / 二创不在库 → 这些自动 fallback 到 VLM。
**风险**：trace.moe 免费档限流 → 缓存（`_cache` 已有）；需先判"这张该不该走 trace.moe"（非动画截图别浪费调用）。

### I2 · 角色识别 → 声优网络
**目标**：截图里的角色是谁 → Bangumi character → 直接串现有 `explore_voice_network`。
**做法**：VLM 出角色候选 → 回锚 `search_characters` 命中后接声优网络 / 角色卡工具链。
**能力边界**：热门角色 OK，冷门 / 撞脸弱 → 给候选 + confidence，让用户确认，不直接断言。

### I3 · 多图 + 置信度校准
**目标**：当前前端单图、后端支持 4 图——补齐多图上传；同时把 VLM 自报的 confidence 与"回锚是否命中"做校准（命中 Bangumi 才升 confidence）。
**做法**：前端多图 UI；后端已支持 attachments 列表；识番结果按"回锚命中"二次加权。

### O1 · 台词 / 字幕 OCR → 定位
**目标**：galgame CG / 漫画截图里的台词 → 检索是哪部、哪个场景。
**做法**：`qwen3.6-flash` OCR（`qwenvl markdown` 提示词，填进 config 的 `vlm_ocr_hint`）抽文本 → 全文检索（萌娘 / wiki / web）回锚作品。
**能力边界**：OCR 本身强；"哪个场景"取决于文本检索质量，给候选不强断言。

### O2 · 榜单 / 杂志 / PPT → 结构化
**目标**：B站新番榜单截图、杂志情报页、活动信息图 → 结构化（作品名 / 时间 / 评分）。
**做法**：`qwenvl html` / `qwenvl markdown` 解析为带位置信息的结构化文本 → 抽实体回锚 Bangumi。
**价值**：直接喂给 V1（无字幕榜单视频），也能独立做"情报图整理"。

### F1 · 画风描述 → 相似作品入口
**目标**：给一张图，描述画风 → 找视觉氛围相近的作品。
**做法**：VLM 出画风 / 色调 / 题材标签 → 映射到 Bangumi tag → 走现有 `recommend_subjects` 当视觉入口。
**能力边界**：画风→tag 是模糊映射，定位"氛围相似"而非精确；标为弱推荐入口。

### S1 · 以图搜图 / 溯源
**目标**：这张图出自哪（同人图 / 插画 / cosplay）。
**做法**：接 **SauceNAO / ascii2d / trace.moe** 以图搜图 → 溯源到 pixiv / 原作 → 标来源。
**风险**：**版权敏感**——只做"溯源 / 标来源"，不提供下载、不存储原图；反爬需限频。

### S2 · pixiv 图像推荐
**目标**：基于画师 / tag / 相似插画做图像推荐。
**现状**：无 pixiv 数据源。
**做法**：pixiv API（需登录态）→ 画师 / tag 推荐；或先做轻量版"画师作品溯源"。
**风险**：**版权与合规是硬门槛**（R18 过滤、登录态、ToS、反爬）→ 排最后，先评估法律边界再动。

### V1 · 抽帧 + OCR（无字幕视频）
**目标**：补 D2 缺口——无字幕的 PPT / 榜单类导视视频（`videos` tool 的 metadata 层 caveat 已标"需抽帧 + OCR/VLM"）。
**做法**：按时间 / 场景切分抽关键帧 → `qwen3.6-flash` OCR + 描述 → 结构化（接 O2）。
**能力边界**：抽帧策略影响质量；成本（每帧一次 VLM）→ 限帧数 + 缓存。

### V2 · 关键帧识番
**目标**：视频片段 → 是哪部番。
**做法**：抽关键帧 → trace.moe / VLM（同 I1 逻辑）。

## 4. 专用引擎对照（VLM 补不了的靠它们）

| 引擎 | 定位 | 与 VLM 的分工 |
|---|---|---|
| **trace.moe** | 动画截图 → 番 + 集 + 秒 | 精确识番主力，VLM 做语义补充 |
| **SauceNAO / ascii2d / IQDB** | 以图搜图 / 同人溯源 | 溯源主力，VLM 做画面描述 |
| **pixiv API** | 插画 / 画师推荐源 | 数据源，VLM 做画风标签 |
| **qwen3.6-flash / Qwen-VL** | OCR / 描述 / 热门识别 | 语义入口、文字提取 |

---

## 5. 与现有架构衔接

- 都套 Tool 契约 + `upload://`（已支持，扩多图）+ 回锚 canonical + source/confidence。
- **新 source 类型**：`image`（VLM 识别）、`image_match`（trace.moe / SauceNAO 溯源）、`pixiv`。
- **claim verifier**：视觉识别 claim 当弱证据（低 confidence），回锚命中后才升级；"识别为 X"标**推理**不标事实——直接复用刚加固的 claim 校验。
- **成本**：识图高频 → 学生券（300 元无门槛 + 先用后返 + 3 个月 Pro 资源包）兜初期；结果走 `_cache`。

---

## 6. 优先级分波

- **第一波（地基 + 亮点，靠 qwen3.6-flash + trace.moe 就能做）**：I1 trace.moe 双路 · I2 角色识别 · O1 台词 OCR
- **第二波（补能力）**：O2 榜单结构化 · V1 抽帧 OCR · F1 画风推荐 · I3 多图
- **第三波（合规 / 外部依赖重）**：S1 以图搜图 · S2 pixiv 推荐 · V2 视频识番

> 每项落地前先确认三件事：**qwen3.6-flash 能做多少 / 要不要专用引擎 / 版权反爬风险**。

---

## 7. 风险与合规

- **版权**：同人图 / pixiv / 视频画面——只溯源标来源，不存储、不分发；R18 内容过滤。
- **反爬**：pixiv / B站——限频 + 缓存 + 优雅降级（沿用 `videos` tool 的 code≠0 抛错思路）。
- **幻觉**：VLM 识别必须回锚，绝不让"识别结果"直接进答案当事实（claim verifier 兜底）。
- **成本**：识图高频 → 缓存 + 学生券；抽帧限数。

---

## 8. 与 RL 的关系（呼应 docs/15）

- 视觉识别 → 回锚 → 用户确认（对 / 错）= **又一条 reward 信号**（识别准确率），且天然有 ground truth（回锚命中与否）。
- 识番 / 角色识别的"候选 → 确认"轨迹可纳入 C1 采集。
- 但视觉 RL 排在文本 Agentic-RL 之后——多模态属 docs/15 的 D 类**能力补全**，不是 RL 地基（地基仍是 A1/B1/C1）。
